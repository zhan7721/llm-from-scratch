## What is Online DPO?

### The Problem with Offline DPO

In standard (offline) DPO, preference pairs are collected once from a fixed policy and reused throughout training. This causes a **distribution shift**: the data was generated by an older version of the policy, but we are training the current version. As the policy improves, the old preference data becomes less relevant.

### Online DPO's Solution: Generate Fresh Data Each Epoch

Online DPO fixes this by generating fresh preference data from the current policy each epoch:

1. **Generate** multiple candidate responses from the current policy
2. **Score** each candidate with a reward model
3. **Select** the best (chosen) and worst (rejected) as a preference pair
4. **Train** with standard DPO loss on the fresh pairs
5. **Repeat** each epoch with new data from the updated policy

The trade-off is computational cost: generating responses is expensive. But the quality of training signal is much better because the preference pairs reflect the current policy's behavior.

## Architecture

### generate_and_score

The core function that creates preference pairs from the current policy:

```python
def generate_and_score(policy, reward_model, prompt, max_new_tokens, num_candidates, temperature):
    # 1. Generate num_candidates responses using temperature sampling
    candidates = []
    for _ in range(num_candidates):
        input_ids = prompt.clone()
        for _ in range(max_new_tokens):
            logits = policy(input_ids)
            next_token = sample(softmax(logits[-1] / temperature))
            input_ids = cat([input_ids, next_token])
        candidates.append(input_ids)

    # 2. Score with reward model
    rewards = reward_model(stack(candidates))

    # 3. Select best/worst
    chosen = candidates[rewards.argmax()]
    rejected = candidates[rewards.argmin()]

    # 4. Compute reference log probs
    chosen_ref_lp = compute_log_probs(policy, chosen)
    rejected_ref_lp = compute_log_probs(policy, rejected)

    return {chosen, rejected, rewards, ref_log_probs}
```

### OnlineDPODataset

Dynamically creates preference pairs from online generation:

```python
dataset = OnlineDPODataset(policy, reward_model, prompts)
# Generates fresh pairs from the current policy
# Can call dataset.refresh() to regenerate after policy updates
```

### OnlineDPOTrainer

Combines generation and training in one loop:

```python
trainer = OnlineDPOTrainer(model, ref_model, reward_model, prompts)
for epoch in range(num_epochs):
    dataset = trainer.generate()  # Fresh preference pairs
    for batch in dataset:
        metrics = trainer.train_step(batch)  # DPO loss
```

### Step 3: OnlineDPOTrainer

Standard DPO training step on fresh data:


## Training Tips

### Hyperparameters

| Parameter | Typical Range | Description |
|-----------|---------------|-------------|
| `beta` | 0.1 - 0.5 | Temperature for DPO loss |
| `lr` | 1e-7 - 5e-6 | Learning rate |
| `num_candidates` | 4 - 16 | Candidates per prompt (more = better signal) |
| `temperature` | 0.7 - 1.2 | Sampling temperature for generation |
| `max_new_tokens` | 32 - 256 | Response length |

### Common Pitfalls

1. **Too few candidates**: With only 2 candidates, the preference signal is noisy. Use at least 4.

2. **Temperature too low**: Makes all candidates similar, reducing the quality of preference pairs.

3. **Temperature too high**: Produces incoherent responses that don't provide useful training signal.

4. **Same-length prompts required**: For batching, prompts should have the same length (or use padding).

## Key Takeaways

1. **Online generation fixes distribution shift.** By generating preference pairs from the current policy each epoch, the training data always reflects the policy's current behavior.

2. **Reward model provides the signal.** Unlike offline DPO where preferences are pre-collected, online DPO uses a reward model to score generated responses and create preference pairs.

3. **More candidates = better signal.** Generating more candidates per prompt gives the reward model more options to distinguish good from bad responses.

4. **Trade-off: compute vs. quality.** Online DPO is more expensive than offline DPO because it generates responses each epoch. But the training signal is much better.

5. **Reference log probs are frozen.** The reference log probs are computed at generation time and stored, not recomputed during training. This is important for stable DPO training.
